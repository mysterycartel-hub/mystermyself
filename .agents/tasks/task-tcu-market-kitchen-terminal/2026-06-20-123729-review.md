# TCU Market Kitchen Terminal — Interactive trading terminal with demo-first architecture

The change replaces the static Chart Kitchen page with a fully interactive terminal built on TradingView Lightweight Charts, adds an AI coach adapter with mock fallback, a pure-computation technical analysis engine, a new dashboard/journal/roadmap, and a Supabase schema. The architecture treats demo mode as the default path: every data layer (market data, AI coach, auth, persistence) gracefully degrades when env vars are missing, so the app renders and functions without any external services. Auth redirects are fixed from `/passport` to `/dashboard`, and no hardcoded localhost references exist.

**Watch for:** Journal page inserts columns that don't exist in the `tcu-schema.sql` definition (confirmed). AI coach API keys are server-only env vars referenced from client code, making live providers unreachable without a server-side proxy (confirmed). The AnalyzeButton silently swallows errors with no user feedback (confirmed).

## High-level view

The market data layer uses a seeded PRNG to generate deterministic demo candles per symbol+timeframe, with real provider adapters (Twelve Data, Polygon) stubbed behind an env-var gate. The chart always renders something meaningful on a fresh clone. The market data API key is passed as a URL query parameter, which exposes it in browser network logs.

The AI coach calls `process.env.OPENAI_API_KEY` and `process.env.ANTHROPIC_API_KEY` from client-side code. Next.js does not expose non-`NEXT_PUBLIC_` env vars to the browser, so the live providers are architecturally unreachable from the current call site. The mock path always runs, satisfying the "demo fallback" requirement — but the live AI code path is dead without a server-side API route.

The journal page persists to localStorage and attempts Supabase cloud sync, but inserts fields (`pair`, `setup`, `result`, `lesson`, `date`) that don't exist in the `journal_entries` schema (which defines `symbol`, `timeframe`, `bias`, `notes`, `ai_summary`). Cloud sync is broken by design until the schema or the code is aligned.

The TCU overlay engine is a pure computation module that derives structure, liquidity zones, and a trade plan from candle arrays. The `!hasCHOCH` confluence factor in setup quality grading penalizes trend-confirming CHOCHs — a simplification that produces counterintuitive results in legitimate trending setups.

## AI coach: dead code path for live providers

```typescript
function getApiKey(provider: AIProvider): string | null {
  if (provider === 'openai') {
    return process.env.OPENAI_API_KEY ?? null  // never available in browser
  }
  if (provider === 'anthropic') {
    return process.env.ANTHROPIC_API_KEY ?? null  // never available in browser
  }
  return null
}
```

Since `AnalyzeButton.tsx` is a `'use client'` component and imports `analyzeChart` directly, this function runs in the browser where non-prefixed env vars are always `undefined`. The code will always fall to mock mode. The project already has an `/api/coach` route (referenced in context.json) — the intended pattern is likely to call that API route instead of the direct provider functions when a real key is configured.

## Market data: API key exposure in URL

The Twelve Data and Polygon provider stubs pass the API key as a URL query parameter (`&apikey=${apiKey}`, `&apiKey=${apiKey}`). Since `fetchCandles` runs client-side (imported in the `'use client'` CandlestickChart component), the key will appear in browser DevTools network tab. This is standard practice for these providers' public-tier keys, but if a server-side key is used (via `MARKET_DATA_API_KEY`), it should be routed through an API endpoint instead.

## TCU overlay engine: CHOCH confluence simplification

The trade plan builder uses a confluence-counting approach to assign setup quality grades:

```typescript
const confluenceCount = [hasLeftover, hasZone, hasTarget, !hasCHOCH].filter(Boolean).length
```

The `!hasCHOCH` factor means any CHOCH in the recent structure breaks _reduces_ setup quality. In a trending market, a CHOCH that confirms the new bias direction is actually a strong signal — penalizing it produces `B` or `C` grades on setups that traders would consider `A` or `A+`. This is worth revisiting when the engine matures.

## Journal schema mismatch (confirmed)

The `journal_entries` table in `db/tcu-schema.sql`:
```sql
CREATE TABLE journal_entries (
  id, user_id, symbol, timeframe, bias, notes, ai_summary, created_at
);
```

The journal page insert in `app/journal/page.tsx`:
```typescript
await supabase.from('journal_entries').insert({
  user_id, date, pair, setup, result, notes, lesson
})
```

Only `user_id` and `notes` exist in both. The insert will fail with a Postgres column-not-found error when Supabase is live. The fallback to localStorage still works, so the app won't crash — but cloud sync is broken until aligned. The `loadEntries` function has the same problem in reverse: it maps columns that don't exist in the actual table.

## Auth callback: redundant origin

```typescript
const { searchParams, origin } = new URL(request.url)  // line 4
// ... later ...
const { origin: reqOrigin } = new URL(request.url)      // line 19
```

Both produce the same value from the same input. The second extraction should use the already-destructured `origin`.
